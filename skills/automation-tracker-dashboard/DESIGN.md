# DESIGN.md — automation-tracker-dashboard

*Design intent. Written 2026-04-22.*

## What this is

A single Obsidian page that materializes the NSLS Automation Tracker. Refreshed daily at 7 AM. Shows: portfolio counts by stage/type, Anish's owned automations with checklist progress, recent activity, SLT/Finance Prototypes to watch, and the full portfolio by department.

## Customers

**Primary:** Anish. Specifically as a motivational + operational artifact — he can see his automation portfolio grow, see which of his automations are close to Production, and see what else in the org is in flight without opening Airtable.

**Indirect:** nobody. Other SLT members could benefit from similar dashboards but this one lives in Anish's vault.

## UX principles

1. **Airtable UI doesn't surface trends; this page does.** The value is making "portfolio state" scannable in 30 seconds, not duplicating Airtable's record view.
2. **Read-only.** Never writes to the Tracker. If a field is wrong, fix it in Airtable.
3. **Deterministic.** Same API state → same dashboard. No randomness, no summarization choices.
4. **Cheap.** 2 HTTP calls, stdlib Python, no LLM, runs in under 5 seconds.
5. **Obsidian-native.** Markdown with YAML frontmatter. Backlinks from daily notes work. Search works. No JS, no HTML dashboard.

## What this should NOT become

- **A live Airtable viewer.** If live is needed, open Airtable — the 5-minute staleness here is acceptable.
- **A Slack bot.** The dashboard is *passive* context, not an alerting system. "The X automation advanced to Production" isn't newsworthy enough for Slack.
- **A cross-builder leaderboard.** Automation tracking is NOT competitive. The SLT leaderboard section is a *relationship* signal (who hasn't shipped lately — check in on them), not a ranking.
- **A stage-advancement gate.** The dashboard does not enforce checklist completion. `/register-automation` does that server-side. The dashboard just surfaces it.

## Interaction surface

- **Channel:** `Obsidian/AP/03-meta/automations-dashboard.md`.
- **Trigger:** Daily 7:00 AM via Windows Task Scheduler.
- **Inputs:** `GET /automations`, `GET /builder-stats/{email}`.
- **Output:** single markdown file, overwritten each run.

## Why this shape

- **Alternatives considered:** a browser extension that overlays Airtable; a Notion mirror; a Slack digest. All rejected because Anish's primary workspace is Obsidian. The tool should meet him where he already reads.
- **Why not generated by the weekly-brief skill?** Scope drift. Weekly-brief synthesizes human patterns. The dashboard shows API state. Different granularity.

## Measuring success

**Green signals:**
- Anish opens the dashboard on most days (Obsidian access timestamps).
- "Ready to advance" automations actually get advanced within a few days of showing up.
- The dashboard reduces Airtable opens for status queries.

**Red flags:**
- Anish ignores the dashboard for >2 weeks → either it's not useful (redesign) or he's too busy (keep running; check back).
- Sections become stale or unused → remove them. A 30-second page beats a 60-second page.

## Guardrails

- Stdlib-only, no external HTTP libs → no supply-chain surprises.
- No writes — the dashboard is incapable of mutating the Tracker even if misused.
- Safe to run repeatedly; no rate-limit concerns at daily cadence.
